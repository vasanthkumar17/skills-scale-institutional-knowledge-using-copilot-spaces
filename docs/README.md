# OctoAcme Project Management Docs

## Overview

This README is the single entry point for all OctoAcme project management process documents. It provides a concise summary of how OctoAcme runs projects so that new and existing team members can quickly understand our approach, discover the right resources, and apply consistent practices across every project.

## Project Management Summary

OctoAcme follows a structured lifecycle—**Initiation → Planning → Execution → Release → Close & Retrospective**—guided by the principles of customer-first iterative delivery, clear ownership, data-informed decisions, and psychological safety. Key artifacts produced at each stage include the Project One-pager/Charter, Roadmap & Release Plan, Sprint/Iteration Backlog, Acceptance Criteria & Definition of Done, Risk Register, and Retrospective notes with action items.

Core roles span the full delivery team: the **Project Manager** coordinates delivery, schedules, risks, and communications; the **Product Manager** owns outcomes, backlog prioritization, and success metrics; **Developers** build, test, and review working software; **QA/Testing** validates acceptance criteria and overall quality; and **Stakeholders** provide inputs and approvals throughout the lifecycle. Communication runs on a regular cadence—daily standups, a weekly delivery sync, and a demo/review each sprint or milestone—with a weekly status template and a single source of truth for project status.

Quality is maintained through small, well-scoped pull requests that include issue links and acceptance criteria in the description, CI pipelines (tests, linting, security scans), required approvals, and a layered test strategy (unit, integration, and end-to-end smoke tests) complemented by manual QA when needed. Releases follow a checklist with a rollback plan, post-deploy verification, and incident communication protocols. Each cycle closes with a blameless retrospective to capture learnings and drive continuous improvement.
